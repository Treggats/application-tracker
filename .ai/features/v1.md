## Enums

### ApplicationStatus (backed enum)
`lead` → `applied` → `interviewing` → terminal: `offer` | `rejected` | `withdrawn`

### InteractionType (backed enum)
`note`, `status_change`, `email`, `call`, `interview`

## Behaviour

- **A status change automatically generates an `Interaction`** of type `status_change`, via an observer on `Application`. The observer keeps `status` (current truth) and the interaction history in sync. This is the only place that may happen — no duplicate write paths.
- `Application::status` is the source for fast queries; the `status_change` interactions are the history.

## Enrichment

Behind an interface, so progress does not depend on an API account.

- `CompanyEnrichment` (interface)
- `NullEnrichment` — v1 default, no external call
- `KvkEnrichment` — optional, official KVK API (developers.kvk.nl, paid per query, account + key required). Fills `kvk_number`, `city`, `sbi_*`, sets `enriched_at`.

The free KVK Open Dataset (HVDS) is unsuitable: it contains no company name/KVK number and is an analysis dataset, not a lookup API. Do not use it for this feature.

Caching/error handling: `enriched_at` determines whether to re-query; on an empty/failed response the existing data stays and is not overwritten. This rule must have a Pest test, not just a spec line.

## Out of scope (explicit)

- Auth / multi-user (no `user_id` on models — do not carry a dead column)
- Tags/skills per role + filtering → v2
- Vacancy integration → v3
- Notifications, reminders, email
- Document/CV uploads per application
- Dashboards/statistics
- JS frontend / component spectacle
